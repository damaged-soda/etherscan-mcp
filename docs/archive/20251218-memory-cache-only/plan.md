# 内存缓存化（去掉文件缓存） 技术变更计划（Plan）

## 0. 背景与目标（简短即可）
- 背景：当前 ContractCache 支持可选文件落盘；用户希望缓存仅存在于进程内存，重启即清空，避免文件读写和持久化。
- 目标：移除/禁用文件缓存路径，缓存仅驻留内存；保持现有接口兼容性和功能正确性。
- 非目标：不引入新存储、不更改 Etherscan 交互逻辑、不引入新的配置项。

## 1. 影响范围（必须）
- 影响的 repo（来自 docmap，可多项）：etherscan-mcp
- 影响的模块/目录/文件（按 repo 分组列出即可）：app/cache.py（去掉磁盘持久化逻辑）、app/service.py（构造 ContractCache 时的参数/行为）、app/config.py（如涉及 cache_dir 配置说明）、SOT 文档。
- 外部可见变化（如适用：API/CLI/配置/数据格式）：缓存不再落盘，重启后缓存丢失；可能不再使用 CACHE_DIR 环境变量。

## 2. 方案与改动点（必须）
说明：实现将按批次推进；每批次写入前需先列出本批次改动文件/影响并征求确认。

- repo: etherscan-mcp
  - 改动点：简化 ContractCache 为纯内存实现；移除/忽略 cache_dir；service 构造时不传磁盘路径；如有文件写入逻辑一并删除；更新配置/文档描述。
  - 新增/修改的接口或数据结构：无对外接口变化；可能保留 cache_dir 参数但不使用，或直接移除字段（需确认兼容性）。
  - 关键逻辑说明：缓存键仍按 address+chainid；进程结束即丢失；Etherscan 请求逻辑不变。

## 3. 自测与验收口径（必须，可执行）
- 本地自测步骤（命令/操作）：运行调用（fetch_contract / call_function），验证缓存命中（第二次不重复请求可通过插桩/日志观察或在无网络情况下读取已有缓存是否失败）；重启进程后再次调用需重新请求。
- 关键用例清单：
  - 同进程重复调用同一地址命中缓存（无磁盘依赖）。
  - 删除 `.cache/etherscan` 目录后功能不受影响（无需读写磁盘）。
  - 重启进程后需重新请求 Etherscan（缓存丢失）。
- 通过标准：上述用例符合预期；无文件写入/读取；API 行为不变。
