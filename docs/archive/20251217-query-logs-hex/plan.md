# query_logs 支持 0x 块高 / 报错提示优化 技术变更计划（Plan）

## 0. 背景与目标（简短即可）
- 背景：当前 MCP 工具 `query_logs` 的 `from_block`/`to_block` 仅接受十进制整数；用户传入以 `0x` 开头的十六进制块高会触发 Pydantic 的原始类型报错，提示不够友好。
- 目标：输入层兼容十六进制块高（含 `0x` 前缀），或在格式错误时给出更明确的引导，避免用户手动转换/踩坑。
- 非目标：不改 Etherscan 请求逻辑、缓存策略、其他工具接口；不引入新的依赖。

## 1. 影响范围（必须）
- 影响的 repo（来自 docmap，可多项）：etherscan-mcp
- 影响的模块/目录/文件（按 repo 分组列出即可）：
  - app.mcp_server 输入模型 / 工具定义
  - 可能的日志查询服务参数校验/转换逻辑
- 外部可见变化（如适用：API/CLI/配置/数据格式）：
  - MCP 工具 `query_logs` 支持 `from_block`/`to_block` 传入十进制或带 `0x` 的十六进制字符串；非法输入时报错提示说明需要十进制或 `0x` 十六进制。

## 2. 方案与改动点（必须）
说明：实现将按批次推进；每批次写入前需先列出本批次改动文件/影响并征求确认。

- repo: etherscan-mcp
  - 改动点：
    - 在 `query_logs` 的请求模型/工具参数层加入块高解析：接受 int、十进制字符串或 `0x` 十六进制字符串，统一转换为 int。
    - 优化错误提示：明确提示可接受的格式，并示例十进制块高。
  - 新增/修改的接口或数据结构：
    - `query_logs` 工具参数解析规则（兼容 0x 十六进制）。
  - 关键逻辑说明：
    - 使用辅助解析函数或 Pydantic validator 检测字符串是否以 `0x` 开头并按 16 进制解析；否则按 10 进制；保留现有 int 输入路径；对非法输入抛出带格式说明的 `ValueError`。
